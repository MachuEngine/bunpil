# Claude Code 빌드 프롬프트 (Phase별)

> 전제: `CLAUDE.md`에 설계 스펙이 로드돼 있음.
> 사용법: Phase 순서대로 아래 블록을 그대로 붙여넣기 → 완료 기준 확인 → `git commit` → 다음 Phase.
> 막히면 "이 단계를 설명하면서 같이 진행해줘"로 학습 모드 전환.

---

## Phase 0 — 프로젝트 스캐폴딩

```
CLAUDE.md 스펙에 따라 프로젝트 뼈대를 만들어줘.
- Python 가상환경(venv) + 의존성 관리(requirements.txt 또는 pyproject)
- FastAPI 비동기 서버, 모듈 분리 구조(common/rag, common/llm, modules/exam, modules/record)
- 환경변수: .env.example(커밋용 플레이스홀더 템플릿) + .env(실값, .gitignore에 추가) 생성. 시크릿 커밋 금지
- Dockerfile + docker-compose(로컬 검증용)
- GET /health 엔드포인트
이 Phase 범위만 작업하고 다음 단계는 건드리지 마.
완료 기준: 가상환경에서 의존성 설치 + `docker-compose up` 후 /health 가 200. 확인되면 멈추고 보고해줘.
```

## Phase 1 — RAG 인프라 (공통)

```
공통 RAG 인프라를 구현해줘.
- PDF 파싱(PyMuPDF) → 청킹 → BGE-M3 임베딩 → ChromaDB 적재
- BGE-reranker로 재정렬
- 메타데이터 태깅: 출처·연도
- 임베딩·리랭킹은 CPU에서 수행(소규모 코퍼스 전제)
- 컬렉션 구분: 영구 컬렉션(공개 자료) vs 세션 한정 임시 컬렉션(업로드 지문, 처리 후 폐기)
샘플 PDF 1개로 인덱싱 + 검색 동작 테스트 포함.
완료 기준: 샘플 쿼리 → rerank된 관련 청크 반환. 확인되면 멈추고 보고해줘.
```

## Phase 2 — LLM 추상화 레이어

```
LLM 호출 추상화 레이어를 만들어줘.
- 공통 인터페이스(generate), 백엔드 교체 가능: 개발=로컬(Ollama/소형 모델), 배포=RunPod 서버리스 엔드포인트

지금은 로컬 백엔드로만 동작시키면 돼.
완료 기준: 프롬프트 → 응답 수신, 백엔드 전환 동작. 확인되면 멈추고 보고해줘.
```

## Phase 3 — 출제 모듈 (ReAct Agent)

```
출제 모듈을 LangGraph ReAct Agent로 구현해줘. CLAUDE.md의 아키텍처를 따른다.
- State: spec(단원·문항수·유형분포·난이도분포·대상) / source / standards(+커버리지 맵) / draft_items / budget
- 노드: 세트 구성 계획 → Agent(ReAct) ↔ 도구 → 구성 충족 검증 → 출력
- 도구: 지문·기준 RAG / 문항 생성(Few-shot) / 품질 검증(LLM Judge) / 중복 검증(기출 유사도)
- Agent는 관찰 기반으로 판단: 특정 문항 재생성 / 난이도 조정 / 커버리지 보충 / 중복 폐기 / 종료
- budget으로 무한루프 방지
- 업로드 지문은 세션 임시 컬렉션으로 처리 후 폐기(Phase 1 규칙)
완료 기준: 단원 지문 업로드 → 제약(유형·난이도·커버리지·비중복) 만족하는 문항 세트 생성. 통합 테스트 포함. 확인되면 멈추고 보고해줘.
```

## Phase 4 — 출제 평가

```
출제 모듈 평가 스크립트를 만들어줘. 데이터는 공개/합성만 사용.
- 검색: Recall@5, MRR (골든셋 ~30, 질의→정답 청크)
- 문항: LLM Judge 루브릭(정답 유일성·오답 매력도·근거성), 5점 척도
- 세트: 분포·커버리지·중복률 함수 검증
- Judge 신뢰도: 사람 라벨 ~30개와 일치율(또는 kappa) 검증 루틴
완료 기준: 평가 스크립트 실행 → 지표 리포트 출력. 확인되면 멈추고 보고해줘.
```

## Phase 5 — 생기부 모듈 (검증 Chain)

```
생기부 윤문 모듈을 LCEL Chain으로 구현해줘. 보안 규칙을 엄격히 지킨다.
- 흐름: 교사 관찰 메모 입력 → 개인정보 마스킹 → 윤문 생성(Few-shot 생기부 문체) → 규정 RAG 검증(위반 플래그) → 출력 + 교사 책임 고지
- 위반 시 윤문 단계로 재시도
- 규정 코퍼스(기재요령) 인덱싱 후 검증에 사용
[엄수] 메모에 없는 사실 추가 금지(생성 아닌 '다듬기'). 마스킹은 외부/모델 호출 전에. 입력은 비저장. 로그에 PII 금지.
완료 기준: 메모 입력 → 마스킹 동작 + 윤문 + 규정 위반 플래그 표시. 확인되면 멈추고 보고해줘.
```

## Phase 6 — 생기부 평가

```
생기부 모듈 평가 스크립트를 만들어줘. 안전 지표 우선, 합성 데이터만.
- 마스킹 누락률(FN) — 목표 0
- 사실 추가율(메모에 없는 내용, NLI식 Judge) — 목표 0
- 규정 위반 검출 Recall/F1 — 목표 Recall ≥ 0.95
- 골든셋: 위반 심은 문장 50 + 마스킹 테스트 20 + 메모→윤문 쌍 20 (합성)
완료 기준: 마스킹 누락 0, 사실 추가 0, 규정 Recall ≥ 0.95 리포트. 확인되면 멈추고 보고해줘.
```

## Phase 7 — UI (Gradio)

```
Gradio로 2모드 UI를 만들어줘.
- 출제 모드: 단원 업로드 + 요청 → 문항 세트 (대화로 수정 가능)
- 생기부 모드: 관찰 메모 입력 → 윤문 + 규정 위반 플래그
- 생기부 출력에 "교사 최종 책임(보조수단)" 고지 표시
- 두 모드 모두 백엔드(Phase 3/5)와 연결
완료 기준: 브라우저에서 두 기능 end-to-end 동작. 확인되면 멈추고 보고해줘.
```

## Phase 8 — 배포

```
배포를 진행해줘. 학습 목적이니 EC2/Caddy 단계는 무엇을 왜 하는지 설명하면서 같이 가자.
(a) RunPod 서버리스: Qwen2.5 7B(vLLM) 핸들러 컨테이너 패키징 → 서버리스 엔드포인트 배포 → LLM 레이어를 이 엔드포인트로 전환
(b) EC2: t3.small 프로비저닝, 보안그룹 설정, Docker로 앱 배포, ChromaDB는 EBS 볼륨
(c) Caddy 리버스 프록시 + HTTPS(+도메인)
(d) billing alarm 설정
[엄수] 로그에 PII 금지 재확인, .env/시크릿 안전 관리(커밋 금지)
완료 기준: 공개 HTTPS URL 접속 → 출제·생기부 두 기능 동작. 확인되면 멈추고 보고해줘.
```

## 진행 원칙

- 한 Phase = 한 커밋. 완료 기준 통과 후에만 다음으로.
- Phase마다 새 환경변수가 생기면 `.env.example`에 항목 추가하고 `.env`에 실값 채우기(LLM 백엔드 키 Phase 2, RunPod 엔드포인트 Phase 8 등).
- 보안 규칙(마스킹·비저장·로그 PII 금지·사실 무첨가)은 매 Phase 자동 적용 — Claude Code가 어기면 즉시 교정.
- 로직 개발은 로컬/소형 모델로, RunPod 서버리스는 Phase 8에서 연결.
- 막히는 단계는 "설명하며 진행"으로 전환해 학습.
